# 시연 프롬프트

순서대로 실행하세요. 각 단계가 이전 단계 위에 쌓입니다.

---

## Step 1: 코드 리뷰 (Skill → 읽기 전용 분석)

```
/review-code
```

> Skill이 코드를 읽고 보안·품질·성능 이슈를 **보고**합니다.
> SQL Injection, 하드코딩 시크릿, XSS 등이 발견됩니다.

**포인트**: "Skill은 분석 결과를 보여주지만, 코드를 수정하지는 않습니다."

---

## Step 2: 테스트 생성 (Skill → 테스트 파일 생성)

```
/gen-test
```

> Skill이 모든 엔드포인트에 대한 테스트 파일을 **생성**합니다.
> 보안 테스트가 포함되어 있어서, 현재 취약한 코드에서는 일부 테스트가 **실패**합니다.

**포인트**: "Step 1에서 발견한 문제를 테스트 코드로 증명합니다."

---

## Step 3: 수정 + 검증 (Agent → 코드 수정 + 테스트 실행)

```
보안 이슈를 수정하고 테스트로 검증해줘.

1. security-agent로 server.py와 db.py의 보안 취약점을 직접 수정해줘
2. test-agent로 기존 테스트를 실행해서 수정이 잘 되었는지 검증해줘
3. 수정 전후 비교를 정리해줘

두 에이전트를 동시에 실행해줘.
```

> security-agent가 코드를 **직접 수정**하고, test-agent가 기존 테스트를 **실행**합니다.
> 수정 전 실패하던 테스트가 수정 후 통과하는 것을 보여줍니다.

**포인트**: "Skill은 분석/생성, Agent는 수정/검증. 역할이 다릅니다."

---

## 역할 비교

| 도구 | 하는 일 | 코드 수정 |
|------|---------|-----------|
| `/review` | 이슈 분석·보고 | X |
| `/gen-test` | 테스트 파일 생성 | 테스트 파일만 |
| security-agent | 취약점 직접 수정 | O (server.py, db.py) |
| test-agent | 기존 테스트 실행·보고 | X |

---

## 시연 팁

- Step 1→2→3 순서를 지키세요. **발견 → 증명 → 해결** 흐름이 스토리를 만듭니다.
- Step 2 후 `cd demo-scenario/src && python -m pytest tests/ -v`로 실패를 라이브로 보여주면 좋습니다.
- Step 3에서 두 에이전트가 동시에 실행되는 것을 강조하세요.
- 최종: 같은 테스트 명령어로 전체 통과를 보여주면 임팩트가 큽니다.
